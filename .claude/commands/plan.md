# /plan - 需求规划命令

## 用途
在开始编码前，先分析需求、识别风险、制定实施计划。

## 使用场景
- 新增功能
- 重构代码
- 修复复杂 Bug
- 架构调整

## 执行流程

### 1. 复述需求（必须）
```
我理解您的需求是：
- 功能点 1：...
- 功能点 2：...
- 预期效果：...

我的理解是否正确？如有偏差请指正。
```

### 2. 分析现状
- 使用 Glob/Grep 查找相关代码
- 阅读关键文件
- 了解现有实现

### 3. 识别风险
列出以下类型的风险：
- **技术风险**：新技术、技术栈限制
- **架构风险**：架构变更、模块耦合
- **依赖风险**：外部依赖、模块依赖
- **数据风险**：数据迁移、数据兼容
- **安全风险**：权限、敏感数据
- **性能风险**：性能瓶颈、资源消耗

### 4. 提出方案
提供 1-3 个可选方案，对比优缺点：

**方案 A：{方案名称}**（推荐）
- 优点：...
- 缺点：...
- 工作量：小/中/大
- 风险：低/中/高

**方案 B：{方案名称}**
- 优点：...
- 缺点：...
- 工作量：小/中/大
- 风险：低/中/高

### 5. 拆解步骤
将推荐方案拆解成具体步骤：

```
## 实施步骤

### 阶段 1：{阶段名称}
- [ ] 步骤 1.1：{步骤描述}
  - 文件：{具体文件路径}
  - 内容：{要做什么}
  - 验收：{如何验证}

- [ ] 步骤 1.2：{步骤描述}
  - 文件：{具体文件路径}
  - 内容：{要做什么}
  - 验收：{如何验证}

### 阶段 2：{阶段名称}
...
```

### 6. 明确依赖
```
## 依赖关系
- 步骤 X 依赖 步骤 Y（必须先完成 Y）
- 步骤 A 和 步骤 B 可并行
```

### 7. 等待确认（必须）
```
## 确认
请确认以上计划：
1. 需求理解是否正确？
2. 方案选择是否合适？
3. 步骤拆解是否清晰？
4. 是否有遗漏或需要调整的地方？

✋ 确认后我将开始实施，未确认前不会修改代码。
```

## 约束规则

### 必须遵守
- ✅ 必须先复述需求，等待确认
- ✅ 必须识别风险
- ✅ 必须提供具体的文件路径
- ✅ 必须明确验收标准
- ✅ 必须等待用户确认后才能开始编码
- ❌ 禁止在未确认前修改代码
- ❌ 禁止跳过风险分析
- ❌ 禁止提供模糊的步骤

### 步骤粒度
- 每个步骤应该是一个独立的、可验证的工作单元
- 步骤不宜过大（> 1 小时工作量应拆分）
- 步骤不宜过小（< 5 分钟的琐碎操作可合并）

### 架构风险

## 示例

### 示例 1：新增功能

```markdown
# 需求分析与实施计划

## 需求理解
我理解您的需求是：
- 在缺陷管理中新增"优先级"字段
- 优先级分为：紧急、高、中、低
- 在列表和详情页显示优先级
- 支持按优先级筛选和排序

我的理解是否正确？

## 现状分析
- 缺陷表：sys_defect
- Domain：SysDefect.java
- Mapper：DefectMapper.java
- Service：DefectService.java
- Controller：DefectController.java
- 前端页面：views/system/defect/index.vue

## 风险识别
### 数据风险
- 需要数据库迁移，为现有数据设置默认优先级

### 架构风险
- 无，符合现有架构

## 方案对比
**方案 A：使用字典表**（推荐）
- 优点：灵活，可动态配置优先级选项
- 缺点：需要关联查询
- 工作量：中
- 风险：低

**方案 B：使用枚举**
- 优点：简单，性能好
- 缺点：修改优先级选项需要改代码
- 工作量：小
- 风险：低

## 推荐方案
使用方案 A（字典表），因为：
1. 系统已有字典管理功能
2. 未来可能需要调整优先级选项
3. 符合系统现有设计模式

## 实施步骤

### 阶段 1：数据库和字典
- [ ] 步骤 1.1：添加字典数据
  - 文件：`sql/dict_data.sql`
  - 内容：插入优先级字典数据（紧急、高、中、低）
  - 验收：字典管理页面可见优先级选项

- [ ] 步骤 1.2：数据库迁移
  - 文件：`src/main/resources/db/migration/h2/V0.6.2__add_defect_priority.sql`
  - 内容：ALTER TABLE sys_defect ADD COLUMN defect_priority VARCHAR(20) DEFAULT 'MEDIUM'
  - 验收：执行迁移后字段存在

### 阶段 2：后端实现
- [ ] 步骤 2.1：更新 Domain
  - 文件：`cat2bug-platform-system/src/main/java/com/cat2bug/system/domain/SysDefect.java`
  - 内容：添加 defectPriority 字段
  - 验收：字段可正常读写

- [ ] 步骤 2.2：更新 Mapper
  - 文件：`cat2bug-platform-system/src/main/resources/mapper/system/DefectMapper.xml`
  - 内容：在 SQL 中添加 defect_priority 字段
  - 验收：CRUD 操作包含优先级字段

- [ ] 步骤 2.3：更新 Service
  - 文件：`cat2bug-platform-system/src/main/java/com/cat2bug/system/service/impl/DefectServiceImpl.java`
  - 内容：添加按优先级筛选和排序逻辑
  - 验收：有单元测试

### 阶段 3：前端实现
- [ ] 步骤 3.1：更新列表页
  - 文件：`cat2bug-platform-ui/src/views/system/defect/index.vue`
  - 内容：添加优先级列、筛选器、排序
  - 验收：列表显示优先级，可筛选排序

- [ ] 步骤 3.2：更新表单
  - 文件：`cat2bug-platform-ui/src/views/system/defect/index.vue`
  - 内容：添加优先级选择器
  - 验收：新增/编辑时可选择优先级

### 阶段 4：测试
- [ ] 步骤 4.1：单元测试
  - 文件：`src/test/java/.../DefectServiceTest.java`
  - 内容：测试优先级筛选和排序
  - 验收：测试通过

- [ ] 步骤 4.2：手动测试
  - 验收：主要流程测试通过

## 依赖关系
- 步骤 2.1 依赖 步骤 1.2
- 步骤 2.2 依赖 步骤 2.1
- 步骤 2.3 依赖 步骤 2.2
- 步骤 3.1、3.2 依赖 步骤 2.3

## 预估影响
- 影响模块：缺陷管理模块
- 数据库变更：是，添加 defect_priority 字段
- API 变更：是，查询接口增加 priority 参数
- 向后兼容性：兼容，现有数据使用默认值

## 确认
请确认以上计划是否合适，确认后我将开始实施。
```
