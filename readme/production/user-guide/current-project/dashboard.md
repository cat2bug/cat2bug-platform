# 仪表盘 [/project/dashboard](/project/dashboard)

仪表盘是项目的数据概览页面，提供项目关键指标的可视化展示，帮助团队快速了解项目状态。

![仪表盘列表](./images/dashboard_list.png)

## 功能概述

仪表盘从上到下依次展示以下内容：

### 项目选择

显示当前项目名称，点击可下拉切换到其他项目。

### 测试用例统计

显示项目中测试用例的总数量。

### 缺陷统计

缺陷统计包含以下维度：

- **总数量** - 项目中的缺陷总数
- **待处理数量** - 包括处理中和驳回状态的缺陷
- **待验证数量** - 等待测试人员验证的缺陷
- **已关闭数量** - 包括已通过和手动关闭的缺陷

### 交付物统计

显示项目中交付物的总数。

### 项目报告统计

显示项目中生成的报告总数。

### 文档统计

显示项目中文档的总数。

### 成员统计

成员统计包含以下维度：

- **项目成员总数** - 项目中的所有成员总数
- **管理员数量** - 项目管理员数量
- **测试数量** - 测试人员数量
- **开发数量** - 开发人员数量
- **外部人员数量** - 外部人员数量

### 缺陷状态变化走势图

展示缺陷状态随时间的变化趋势，支持切换查看：

- **30天走势** - 最近30天每天缺陷状态的统计趋势
- **12个月走势** - 最近12个月每月缺陷状态的统计趋势

### 成员处理缺陷走势图

展示成员处理缺陷的趋势，支持切换查看：

- **30天走势** - 最近30天每个成员每天处理缺陷的统计趋势
- **12个月走势** - 最近12个月每个成员每月处理缺陷的统计趋势

### 测试计划统计

点击统计标题可下拉切换不同的测试计划，查看该计划的详细统计数据，其中包括：

#### 测试用例统计
- **通过** - 测试通过的用例数量
- **未通过** - 测试未通过的用例数量
- **未执行** - 尚未执行的用例数量
- **总数量** - 该计划中的用例总数

#### 缺陷统计指标
- **缺陷总数** - 该计划发现的缺陷总数
- **[缺陷发现率](#缺陷发现率)** - 测试发现缺陷的比率
- **[缺陷修复率](#缺陷修复率)** - 缺陷被修复的比率
- **[缺陷探测率](#缺陷探测率)** - 缺陷被探测到的比率
- **[缺陷严重率](#缺陷严重率)** - 严重缺陷占比
- **[缺陷重开率](#缺陷重开率)** - 缺陷被重新打开的比率
- **[缺陷逃逸率](#缺陷逃逸率)** - 缺陷逃逸到生产环境的比率
- **[缺陷密度](#缺陷密度)** - 单位代码量的缺陷数量
- **[缺陷修复平均时长](#缺陷修复平均时长)** - 缺陷从发现到修复的平均时间

::: tip 提示
所有指标计算公式可通过鼠标停放到指标上，通过气泡提示显示。点击指标名称可跳转到详细说明。
:::

### 成员处理缺陷排行

展示成员处理缺陷的排行榜，包括：

- **今日排行** - 当天成员处理缺陷的数量排行
- **总排行** - 累计成员处理缺陷的数量排行

### 测试计划燃尽图

展示测试计划的30天燃尽图，点击标题可切换不同的测试计划查看。

燃尽图显示测试计划中剩余工作量随时间的变化趋势，帮助团队了解测试进度。

### 活动记录

按时间降序展示项目中的最新活动记录，包括：

- 缺陷的创建、修改、状态变更等操作
- 测试计划的创建、修改等操作
- 测试用例的创建、修改、执行等操作

**功能说明：**
- 活动记录按时间分组显示
- 点击每条记录可跳转到对应的详情页面
- 目前只显示最新的100条活动记录

## 权限说明

所有项目成员都可以查看仪表盘，但看到的数据范围可能因角色权限而异。

## 指标说明

### 缺陷发现率

#### 定义

衡量测试工作效率的重要指标，它表示测试用例在发现软件中缺陷的比例。

软件测试过程中由测试者发现的缺陷数量与最终由客户发现的缺陷数量之间的比例。比率越高，说明测试的质量越高。

缺陷发现率是衡量测试工作效率的指标，而不是软件开发质量的指标。如果测试中发现的缺陷比较多，这可能说明测试做得比较好，但并不能直接说明软件开发做得比较差。

#### 公式

缺陷发现率 = (∑ 已发现的缺陷总数 / ∑ 已执行的测试用例总数) × 100%

::: tip 注意
一个高的缺陷发现率通常被视为测试质量的一个积极指标。为了提高缺陷发现率，可以采取提高测试覆盖率的策略，确保测试用例能够覆盖软件中的所有功能、模块和子系统。
:::

### 缺陷修复率

#### 定义

缺陷修复率是一个非常重要的指标，用于衡量软件开发过程中缺陷修正的效率和质量。

在软件开发过程中，缺陷修复率是一个非常重要的指标，因为它可以帮助团队了解他们在发现和修复缺陷方面的表现。一个高缺陷修复率意味着团队在发现和修复缺陷方面做得很好，而一个低缺陷修复率则可能意味着团队需要改进其开发和测试流程。

为了计算缺陷修复率，首先需要确定一个时间段，比如一个迭代周期或一个发布周期。然后需要收集以下数据：

1. **已修复的缺陷总数** - 在这个时间段内，团队修复的所有缺陷的数量
2. **已发现的缺陷总数** - 在这个时间段内，团队发现的所有缺陷的数量，包括通过测试、代码审查、用户反馈等方式发现的缺陷

#### 示例

如果一个团队在一个迭代周期内修复了 20 个缺陷，而总共发现了 25 个缺陷，那么他们的缺陷修复率为：

(20 / 25) × 100% = 80%

这意味着这个团队在这个迭代周期内的缺陷修复率为 80%。

#### 公式

缺陷修复率 = (∑ 已修复的缺陷总数 / ∑ 已发现的缺陷总数) × 100%

#### 指标意义

**高缺陷修复率**意味着团队在发现和修复缺陷方面做得很好。这可能是因为团队有一个很好的测试流程，能够在早期发现缺陷并及时修复它们。另外，高缺陷修复率也可能意味着团队在代码审查和质量控制方面做得很好，能够在开发过程中避免一些常见的缺陷。

**低缺陷修复率**可能意味着团队需要改进其开发和测试流程。这可能是因为团队在测试方面做得不够充分，导致一些缺陷在发布后才被发现。另外，低缺陷修复率也可能意味着团队在代码审查和质量控制方面做得不够好，导致一些常见的缺陷没有及时被发现和修复。

因此，团队应该关注他们的缺陷修复率，并且努力提高它。

### 缺陷探测率

#### 定义

DDP (Debug Detect Percent) 是一个衡量测试工作效率的软件质量成本指标。缺陷探测率越高，也就是测试者发现的错误越多，发布后客户发现的错误才可能越少，降低了外部故障不致成本，达到了节约总成本的目的，可获得较高的测试投资率（ROI）。因此，DDP 是衡量测试投资回报的一个重要指标。

测试周期越长，缺陷探测率不一定会越高，但测试成本越高是一个衡量软件测试效率和质量的重要指标。它用于评估测试人员或团队在测试过程中发现缺陷的能力。

#### 公式

缺陷探测率 = (∑ 测试人员发现的缺陷总数 / (∑ 测试人员发现的缺陷总数 + ∑ 外部人员发现的缺陷总数)) × 100%

### 缺陷严重率

#### 定义

缺陷严重率是指测试人员提交缺陷时记录缺陷的严重程度，统计严重级别以上的缺陷数量，与总缺陷数量的比例。它一定程度上可以反映出软件产品的质量。

缺陷严重率更大的作用是可以帮助软件开发团队更好地管理缺陷，优先解决重要的缺陷，提高软件的质量和用户体验。同时，缺陷严重率的监测也可以提高开发团队的效率，在时间和资源有限的情况下，尽可能地修复对软件质量和用户体验影响较大的缺陷，使其更加专注于关键的开发任务，提高软件开发的整体效率。

#### 公式

缺陷严重率 = (∑ 优先级为高的缺陷数量 / ∑ 已发现的缺陷总数) × 100%

### 缺陷重开率

#### 定义

缺陷重开率指的是在软件开发过程中，已经修复的缺陷再次出现的数量与修复的缺陷数量的比例。它的作用是帮助评估开发团队和测试团队的能力，提高效率。

如果缺陷重开率较高，会造成缺陷修复时间增加，导致软件的整个开发周期变长。它可能说明了开发团队修复缺陷能力存疑，或者测试团队提交缺陷的质量存疑，或者开发和测试之间沟通效率存疑。通过监控和降低缺陷重开率，减少维护和修复成本，提高生产效率。

#### 公式

缺陷重开率 = (∑ 被重新开启或驳回的缺陷数量 / ∑ 已修复的缺陷总数) × 100%

### 缺陷逃逸率

#### 定义

缺陷逃逸率指在软件开发过程中，已经交付到客户或实际使用中，但后续还发现了缺陷的比例。它可以通过统计测试过程中发现的缺陷数量和外部人员（客户）反馈的缺陷数量来计算。

缺陷逃逸率可以用来评估软件开发团队和测试团队的质量水平，以及确定项目后续的改进方向。缺陷逃逸率越低，说明团队的质量水平越高，项目的稳定性和可靠性也越好。而如果缺陷逃逸率较高，则需要对团队的工作流程和质量管理进行改进，以提高软件质量和客户满意度。

#### 公式

缺陷逃逸率 = (∑ 外部人员（客户）反馈或实际应用中发现的缺陷数量 / ∑ 已发现的缺陷总数) × 100%

### 缺陷密度

#### 定义

缺陷密度是交付物（软件/模块）在特定运行或开发期间确认的缺陷数量除以交付物（软件/模块）的大小。

缺陷密度标准是指在软件测试过程中，对于每个功能点允许存在的缺陷数量。这个标准通常由开发团队和测试团队共同制定，以确保软件质量和可靠性。

#### 公式

缺陷密度 = ∑ 缺陷总数 / ∑ 交付物数量

### 缺陷修复平均时长

#### 定义

用于描述研发修复线下缺陷的平均时长，这个指标体现了研发解决线下缺陷的效率。

#### 公式

缺陷修复平均时长 = ∑ 处理缺陷所用的总小时数 / ∑ 已发现的缺陷总数
