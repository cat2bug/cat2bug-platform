# 交付物管理 [/project/module](/project/module)

交付物是项目中最终交付给客户的目标或成果，在软件中，代表某个系统、手机 APP、微信小程序。

多个交付物可以以父子上下级的结构体现，如某某系统下可以关联多个模块。在 Cat2Bug-Platform 中，交付物会关联到[测试用例](./case.md)和[缺陷](./defect.md)中。

在交付物管理页面，可以对其进行增删查改操作，如下图：

![创建交付物](./images/deliverable-create.png)

## 什么是交付物

在 Cat2Bug-Platform 中，交付物是组织测试工作的重要维度，它代表了项目中需要测试的具体对象。

**交付物的作用：**

- 组织测试用例的结构
- 关联缺陷到具体模块
- 统计各模块的测试情况
- 追踪各模块的质量状态

## 交付物结构

多个交付物可以以父子上下级的结构体现，如某某系统下可以关联多个模块。

**示例结构：**

```
电商系统
├── 用户模块
│   ├── 用户注册
│   ├── 用户登录
│   └── 个人中心
├── 商品模块
│   ├── 商品列表
│   ├── 商品详情
│   └── 商品搜索
└── 订单模块
    ├── 购物车
    ├── 下单
    └── 订单管理
```

## 创建交付物

在交付物管理页面，可以对其进行增删查改操作。

### 创建步骤

1. **进入交付物管理页面**
   - 在项目菜单中选择【交付物管理】

2. **点击创建按钮**
   - 点击【新建交付物】按钮

3. **填写交付物信息**
   - **交付物名称**（必填）- 简洁明了的名称
   - **交付物描述** - 详细描述交付物的功能和范围
   - **父级交付物** - 选择上级交付物（可选）
   - **负责人** - 选择负责该交付物的成员
   - **交付物类型** - 系统、模块、功能等

4. **保存交付物**
   - 点击【确定】按钮完成创建

### 快速创建

在创建测试用例或缺陷时，如果需要的交付物不存在，可以通过下拉组件中的【添加】功能快速创建交付物，无需跳转到交付物管理页面。

## 修改交付物

### 修改基本信息

1. 在交付物列表中找到要修改的交付物
2. 点击【编辑】按钮
3. 修改交付物信息
4. 点击【确定】保存

### 调整交付物层级

可以通过拖拽方式调整交付物的层级关系：

1. 在交付物树中选中要移动的交付物
2. 拖拽到目标位置
3. 系统自动保存新的层级关系

## 删除交付物

### 删除条件

删除交付物前需要确认：

- 该交付物下没有关联的测试用例
- 该交付物下没有关联的缺陷
- 该交付物下没有子交付物

### 删除步骤

1. 在交付物列表中找到要删除的交付物
2. 点击【删除】按钮
3. 确认删除操作

**注意：** 如果交付物下有关联数据，需要先处理这些数据（删除或转移到其他交付物）才能删除。

## 交付物统计

### 用例统计

查看每个交付物关联的测试用例数量：

- 总用例数
- 各级别用例分布（P0、P1、P2、P3）
- 用例覆盖率

### 缺陷统计

查看每个交付物的缺陷情况：

- 总缺陷数
- 各状态缺陷分布（待处理、处理中、已解决、已关闭）
- 各优先级缺陷分布（紧急、高、中、低）
- 缺陷密度

### 质量评估

根据统计数据评估交付物的质量：

- **优秀** - 缺陷少，用例覆盖充分
- **良好** - 缺陷在可控范围内
- **一般** - 存在一些问题，需要关注
- **较差** - 缺陷较多，需要重点测试

## 交付物与其他模块的关系

### 与测试用例的关系

- 每个测试用例必须关联一个交付物
- 通过交付物可以筛选相关的测试用例
- 交付物的层级结构帮助组织用例

### 与缺陷的关系

- 每个缺陷必须关联一个交付物
- 通过交付物可以统计各模块的缺陷情况
- 便于定位问题集中的模块

### 与项目的关系

- 交付物属于某个项目
- 一个项目可以有多个交付物
- 交付物的结构反映了项目的架构

## 最佳实践

### 交付物命名建议

1. **使用清晰的名称**
   - 避免使用缩写
   - 使用业务术语
   - 保持命名一致性

2. **合理的层级深度**
   - 建议不超过 4 层
   - 每层的粒度要合适
   - 避免过度细分

3. **示例命名**
   ```
   好的命名：
   - 用户管理模块
   - 商品搜索功能
   - 订单支付流程

   不好的命名：
   - 模块1
   - test
   - 功能A
   ```

### 交付物组织建议

1. **按功能模块划分**
   ```
   系统
   ├── 用户模块
   ├── 商品模块
   ├── 订单模块
   └── 支付模块
   ```

2. **按系统层次划分**
   ```
   系统
   ├── 前端
   │   ├── Web端
   │   └── 移动端
   └── 后端
       ├── API服务
       └── 数据库
   ```

3. **按业务流程划分**
   ```
   系统
   ├── 注册流程
   ├── 登录流程
   ├── 购物流程
   └── 支付流程
   ```

### 维护建议

1. **定期 review**
   - 检查交付物结构是否合理
   - 删除不再使用的交付物
   - 合并重复的交付物

2. **保持同步**
   - 交付物结构应与系统架构保持一致
   - 系统架构变更时及时更新交付物
   - 新增功能时及时创建对应交付物

3. **统一管理**
   - 指定专人负责交付物的维护
   - 制定交付物命名规范
   - 定期培训团队成员

## 常见问题

### Q: 交付物和模块有什么区别？

A: 在 Cat2Bug-Platform 中，交付物就是模块的概念，两者是同一个东西。我们使用"交付物"这个术语是为了更贴近项目管理的语境。

### Q: 可以修改交付物的父级吗？

A: 可以。通过拖拽方式可以调整交付物的层级关系，或者在编辑时重新选择父级交付物。

### Q: 删除交付物会影响已有的测试用例和缺陷吗？

A: 如果交付物下有关联的测试用例或缺陷，系统会阻止删除操作。需要先处理这些关联数据（删除或转移到其他交付物）才能删除交付物。

### Q: 如何快速找到某个交付物？

A: 可以使用搜索功能，输入交付物名称的关键字进行搜索。也可以通过展开/折叠树形结构来浏览。

### Q: 交付物的层级有限制吗？

A: 理论上没有层级限制，但建议不超过 4 层，以保持结构清晰。层级过深会增加管理复杂度。
