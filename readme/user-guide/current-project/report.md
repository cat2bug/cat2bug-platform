# 报告管理 [/project/report](/project/report)

报告是对团队、项目、测试用例、缺陷、交付物等工作管理做出的总结性报表，目前可通过 Open API 方式创建。

## 报告列表

报告列表中展示了已经提交的报表信息，点击某一个报告后可查看报告详情，报告可通过 Markdown 格式展示，如下图：

![报告列表](./images/report_list.png)

**列表功能：**

- **筛选功能** - 按时间、类型、创建人筛选报告
- **搜索功能** - 快速搜索报告标题
- **排序功能** - 按创建时间、更新时间排序
- **批量操作** - 批量删除、导出报告

## 报告详情

报告可通过 Markdown 格式展示，支持丰富的内容格式，如下图：

![报告详情](./images/report_details.png)

**报告内容包括：**

### 1. 测试概述

- 测试目标
- 测试范围
- 测试时间
- 测试人员
- 测试环境

### 2. 测试执行情况

- 测试用例总数
- 已执行用例数
- 通过用例数
- 失败用例数
- 阻塞用例数
- 用例执行率
- 用例通过率

### 3. 缺陷统计

- 缺陷总数
- 各状态缺陷分布
  - 待处理
  - 处理中
  - 已解决
  - 已关闭
- 各优先级缺陷分布
  - 紧急
  - 高
  - 中
  - 低
- 各类型缺陷分布
  - 功能缺陷
  - UI 缺陷
  - 性能缺陷
  - 安全缺陷

### 4. 模块质量分析

- 各交付物的缺陷数量
- 各交付物的用例覆盖率
- 高风险模块识别
- 质量趋势分析

### 5. 测试结论

- 测试完成情况
- 质量评估
- 遗留问题
- 发布建议
- 风险提示

### 6. 改进建议

- 测试过程中发现的问题
- 流程改进建议
- 工具优化建议
- 团队协作建议

## 报告导出

可将报告导出为 Word 或 PDF 格式，点击报告详情右上角的导出按钮，如下图：

![报告导出](./images/report_details.png)

### 导出步骤

1. 打开报告详情页
2. 点击右上角的【导出】按钮
3. 选择导出格式
   - **Word 格式** - 适合编辑和打印
   - **PDF 格式** - 适合分享和展示
4. 等待生成完成
5. 下载导出文件

### 导出格式说明

**Word 格式特点：**
- 可以继续编辑
- 支持添加批注
- 适合内部流转
- 文件较大

**PDF 格式特点：**
- 格式固定，不可编辑
- 跨平台兼容性好
- 适合正式发布
- 文件较小

## 创建报告

### 通过 Open API 创建

目前报告主要通过 Open API 方式创建，可以自动化生成测试报告。

**API 调用示例：**

```bash
POST /api/report/create
Content-Type: application/json
X-API-Key: your-api-key

{
  "projectId": 1,
  "reportName": "V1.0 版本测试报告",
  "reportType": "测试报告",
  "content": "# 测试报告内容（Markdown格式）",
  "startTime": "2024-01-01",
  "endTime": "2024-01-15"
}
```

详细的 API 文档请参考 [API 文档](./api.md)。

### 手动创建（规划中）

未来版本将支持在界面上手动创建和编辑报告，提供更友好的编辑体验。

## 报告统计

### 报告趋势分析

查看历史报告数据，分析质量趋势：

- **缺陷趋势** - 缺陷数量随时间的变化
- **通过率趋势** - 用例通过率的变化
- **模块质量趋势** - 各模块质量的变化
- **团队效率趋势** - 测试效率的变化

### 报告对比

对比不同版本或不同时期的报告：

- 缺陷数量对比
- 用例覆盖率对比
- 质量指标对比
- 改进效果评估

## 最佳实践

### 报告编写建议

1. **客观准确**
   - 基于真实数据
   - 避免主观臆断
   - 数据来源可追溯

2. **重点突出**
   - 突出关键问题
   - 标注高风险项
   - 明确改进方向

3. **结论明确**
   - 明确测试是否通过
   - 说明是否可以发布
   - 给出具体建议

4. **图文并茂**
   - 使用图表展示数据
   - 添加截图说明问题
   - 保持格式统一

### 报告审核建议

1. **数据核对**
   - 核对统计数据是否准确
   - 检查计算公式是否正确
   - 确认数据来源可靠

2. **内容完整**
   - 检查是否包含所有必要内容
   - 确认结论是否明确
   - 验证建议是否可行

3. **格式规范**
   - 检查格式是否统一
   - 确认图表是否清晰
   - 验证链接是否有效

### 报告归档建议

1. **及时归档**
   - 测试完成后及时生成报告
   - 报告审核通过后立即归档
   - 定期整理历史报告

2. **分类存储**
   - 按项目分类
   - 按版本分类
   - 按时间分类

3. **权限管理**
   - 设置查看权限
   - 控制导出权限
   - 保护敏感信息

## 常见问题

### Q: 如何生成测试报告？

A: 目前主要通过 Open API 方式生成报告。可以编写脚本自动收集测试数据，调用 API 创建报告。未来版本将支持在界面上手动创建。

### Q: 报告可以修改吗？

A: 已生成的报告可以修改。但建议报告审核通过后不要随意修改，以保持数据的真实性和可追溯性。

### Q: 报告导出失败怎么办？

A: 可能的原因：
- 报告内容过大，导出超时
- 报告格式有误，无法转换
- 网络问题，导出中断

解决方法：
- 简化报告内容
- 检查 Markdown 格式
- 重试导出操作

### Q: 如何自动化生成报告？

A: 可以通过以下方式实现自动化：
1. 使用 Open API 获取测试数据
2. 编写脚本生成 Markdown 格式的报告内容
3. 调用 API 创建报告
4. 可以集成到 CI/CD 流程中

### Q: 报告可以分享给外部人员吗？

A: 可以。可以将报告导出为 PDF 格式，通过邮件或其他方式分享。也可以给外部人员分配"外部人员"角色，让他们在系统中查看报告。
